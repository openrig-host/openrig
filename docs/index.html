Getting Started & Stage Preparation

OpenRig is an open-source tool built specifically for live stage use. While designed for maximum stability, we recommend thorough preparation before performing:

  • Run it through rehearsals with your actual songs, plugins, and MIDI hardware.
  • Test your song switches, scene changes, and panic button under live playing conditions.
  • Always maintain a backup plan (e.g., a hardware fallback or second system) for critical performances.

What is OpenRig?

OpenRig is a live VST3 host that looks, feels, and acts like a digital hardware mixer. No virtual cable spaghetti, no confusing node graphs—just a clean, linear rack of channel strips designed for playing keyboards live.

It's built around the simple realities of keyboard performance:

01

Intuitive Mixer-Based Layout

Your plugins live in vertical channel strips with physical faders, mutes, routing, and bypass controls you already understand.

02

FOH and IEM are Not the Same Mix

Every channel strip has independent faders sending to Front of House (FOH) and In-Ear Monitors (IEM) simultaneously.

03

Seamless Song Switches

Next songs preload on a background worker thread and transition atomically, leaving your current sound active until the swap is complete.

04

Interactive Note-Range & CC Learn

Restrict channels to key zones or map physical knobs using a simple "Play note" or "Wiggle knob" learn mode.

05

A Panic Button That Works

A dedicated panic utility kills all active MIDI notes inside the very next audio block to silence runaways instantly.

Features

Everything you need for a reliable, repeatable live rig.

The Rack

Linear, fixed slot layout

One slot per instrument, in a known order. No clicks, no surprises, no patch-graph archaeology.

Dual-bus mixing on every slot

Independent FOH and IEM levels, mutes, and enables per slot.

Per-slot MIDI channel routing

Global default + per-slot override. Send a CC to one instrument and only that instrument.

Per-instrument stacking

Multiple instances of the same plugin in a slot, with individual level, note range, and enable.

Boutique dark-mode UI

Skeuomorphic knobs, glassy panels. No virtual-cable spaghetti.

Songs, Scenes & Setlists

Songs (Rigs)

A complete patch: plugins, states, channel strips, CC maps, levels. Saved as versioned JSON.

Scenes

Variations within a song (verse / chorus / bridge). Snapshot, recall, MIDI-trigger them.

Setlists

Ordered queue of songs with one-click load and automatic preload of the next rig on a worker thread.

Atomic transitions

New rig is built and validated off-thread, then swapped under lock. A failed build never touches the live rig.

Rollback-by-construction

If the build fails, the current rig keeps playing. No half-loaded songs, ever.

MIDI

Arm-then-wiggle CC learn

Click Learn, move the physical control, it's bound. Range, min/max, invert all supported.

Scene MIDI triggers

Assign a Program Change + channel to any scene. Your hardware sequencer calls the song.

MIDI monitor

See what your keyboard is actually sending, with learn-capture overlay.

MIDI remote settings

Define a hardware device, map its controls once, use them across all rigs.

DSP & Effects

Per-slot channel strip

Gate, parametric EQ, compressor, chorus, reverb. Toggleable per slot.

Sampler + waveform splice editor

Load a WAV, set root note, trim with sample-accurate handles.

Arpeggiator

Up / Down / Up-Down / Random patterns, 0–4 octaves, gate time, BPM to 1 decimal.

Octave harmonizer

Generate sub/up octaves with mode shaping.

MIDI effects

Transposer and friends, per slot.

Master FOH & IEM FX buses

Global reverb / EQ / compression before the outputs.

Live Reliability

Panic button

Sends All Notes Off to every plugin in the next audio block. Hardware-fail-safe.

Atomic JSON persistence

Every save writes to a temp file, renames atomically, and keeps a .bak. Crashes during save cannot corrupt your library.

SEH-protected message loop

If a misbehaving VST3 crashes, the engine survives and the show goes on.

Per-plugin exception isolation

One plugin throwing inside processBlock cannot kill the audio thread. The slot mutes; the rest of the rig keeps playing.

Versioned rig format with migration

v1 rigs auto-upgrade to v2. Unknown future versions are refused, not silently misinterpreted.

How It Works

The signal path, and the song-switch sequence.

Signal flow

                          Keyboard (any MIDI controller)
                                    │  MIDI
                                    ▼
        ┌────────────────────────────────────────────────┐
        │               OpenRigEngine                    │
        │   ┌─────┬─────┬─────┬─────┬─────┬─────┐      │
        │   │ S0  │ S1  │ S2  │ S3  │ S4  │ ... │      │  ← Linear rack
        │   │ Mon │Kbd  │Organ│VSTi │Aux  │     │      │     (no cables)
        │   └─────┴─────┴─────┴─────┴─────┴─────┘      │
        │         │ FOH       │ IEM                      │  ← Dual bus
        │         ▼           ▼                          │
        │   Master FOH    Master IEM                     │
        │     FX bus        FX bus                       │
        └─────────┬──────────────┬───────────────────────┘
                  │              │
                  ▼              ▼
              FOH Out        IEM Out

Switching a song

  1. SetlistManager calls RigTransitioner::transitionToFile(nextSong).
  2. RigBuilder runs on a worker thread: instantiate any new plugins, restore state on reused ones, validate with a silent processBlock.
  3. The transitioner takes the callback lock, swaps pointers only (no allocations, no state restore on the audio thread), and signals the message loop.
  4. The old rig is unloaded. New rig is live. Total switch time: typically 200–800 ms; the loading overlay covers it.
  5. If step 2 failed, the lock is never taken. The current rig keeps playing. You will not know anything happened, by design.

Build from source

Requires Visual Studio 2022+ with the C++ desktop workload, and JUCE 8 at C:/JUCE.

git clone https://github.com/davewaugh/davecore
cd davecore
cmake -B build -DJUCE_ROOT=C:/JUCE
cmake --build build --config Release

Data location

All user data lives under %APPDATA%/OpenRig/:

%APPDATA%/OpenRig/
├── songs/       Rig files (.json)
├── sets/        Setlists
├── backups/     Auto-rotated .bak of rigs
└── settings/    User preferences, MIDI maps

Development & Open Source

Built to be extended, customized, and owned.

OpenRig was built using a modern, collaborative approach—pair-programming directly with AI coding assistants (including Antigravity, Gemini, and Claude) to build features rapidly. The result is a highly tailored, utility-focused application that addresses the specific stage needs of live keyboardists.

Because it's fully open source, you have absolute sovereignty over your rig. If a specific plugin behaves unexpectedly, or if you need to build custom MIDI routing behaviors for your hardware: the source is right there. You can clone the project, tweak the logic, compile a release, and run your custom version on stage.

Philosophy

A few principles drove every design decision.

  1. Predictability over flexibility. A stage rig is not a sketchbook. Slots are fixed. Routing is fixed. You know exactly what sound comes out of which key, every time.
  2. The worst-case failure must be a no-op. A song switch that fails should leave the current rig playing. A plugin crash should leave the other slots playing. A panic button must work in the next audio block, always.
  3. Background work, foreground feel. Plugin loading, preloading, MIDI mapping, persistence — none of it blocks the audio thread, and as little as possible blocks the message thread.
  4. Own your data. Rigs are JSON. Backups are automatic. There is no cloud, no account, no telemetry. Your setlist is a folder of files you can read, edit, version-control, and back up with any tool.
